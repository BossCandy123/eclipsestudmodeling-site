The workflow is designed to distinguish user behavior, provider behavior, and platform failures before money moves.

1. Track the click

Create an offer attempt before redirect with click ID, session ID, provider, device, and risk context.

2. Wait for trusted confirmation

Provider events can move value to pending, but nothing becomes withdrawable until the platform can defend it.

3. Route disputes with proof

Only ask for evidence when it can change the decision, and preserve every review action in the audit trail.

4. Gate payout approval

Unresolved disputes, reversal windows, and duplicate-account risk stop payout release before trust is lost.

    Users need clear answers before they open a ticket.

    The site should explain what is verified, what is still waiting on a provider, and when support actually needs proof from the user.

    Why are points pending instead of available?

    Pending means the platform has a tracked attempt or partial provider signal, but not enough trusted confirmation to make points withdrawable yet.

    When should a user open a dispute?

    Only when the provider delay is abnormal, the provider rejected the completion, duplicate risk blocked approval, or the site detected its own tracking issue.

    What proof should support ask for?

    Only evidence that can change the decision: completion time, screenshot, email receipt, or app or game user ID depending on the offer type.

    What happens if a provider reverses credit later?

    The reversal should subtract from the correct balance bucket, preserve the original credit event, and show the user an explicit reversal state.
